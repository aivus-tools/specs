# Domain Docs

Как скиллы должны читать доменную документацию Aivus перед исследованием кода.

## Перед исследованием прочитать

- **`CONTEXT.md`** в корне репозитория, если есть. Создаётся лениво через `/grill-with-docs`, когда термины реально проясняются. Если нет — работать молча, не флагать отсутствие.
- **`docs/adr/`** — ADR по затрагиваемой области, если есть.
- Источники истины проекта: `Specs/PRODUCT_VISION.md` (видение и роадмап), `Specs/DEV_PROCESS.md` (как ведём разработку, баги и трекинг в ClickUp), `Specs/ARCHITECTURE.md` (стек, модели, API, AI-пайплайн), `Specs/DEPLOYMENT.md`, `Specs/ENV_VARIABLES.md`. Эти файлы уже существуют и описывают актуальное состояние.

## Структура

Single-context репозиторий. `CONTEXT.md` и `docs/adr/` — в корне. `docs/agents/` — символлинк на `Specs/agents/` (конфиг трекера и эта домен-карта версионируются в Specs).

## Использовать словарь проекта

Когда вывод называет доменное понятие (заголовок задачи, предложение по рефактору, имя теста), использовать термин как он определён в `CONTEXT.md` и в `Specs/*`. Не уходить в синонимы. Если понятия нет в словаре — это сигнал: либо изобретается язык, которого в проекте нет (пересмотреть), либо реальный пробел (отметить для `/grill-with-docs`).

## Флагать конфликты с ADR

Если вывод противоречит существующему ADR — явно это назвать, а не молча переписывать.
