# 🏗️ ЭТАП 0: АРХИТЕКТУРА И МИГРАЦИЯ BACKEND

**📅 Длительность:** 3 недели (13 октября - 31 октября)

## Что будет сделано:

### 1. Django Backend - Базовая инфраструктура
**Задачи:**
- Создание Django проекта с правильной структурой приложений
- Настройка PostgreSQL подключения и переменных окружения
- Docker контейнеризация для локальной разработки и production
- Создание всех моделей данных:
  - Users (пользователи с ролями)
  - Vendors & Clients (компании)
  - Categories & Entries (справочники для смет)
  - Briefs (проектные брифы)
  - Offers (коммерческие предложения)
  - Rates (тарифные карточки)
  - Units (единицы измерения)

**Зачем:**
- ✅ Команда Python может поддерживать и развивать backend
- ✅ Прочная база данных для всего приложения
- ✅ Type-safe модели с Django ORM

### 2. API Endpoints (полная копия текущих)
**Задачи:**
Разработка всех REST API endpoints точно совместимых с текущим фронтендом:

**Authentication:**
- `POST /api/v1/auth/register` - регистрация
- `POST /api/v1/auth/login` - вход (email/password + Google OAuth)
- `POST /api/v1/auth/check-email` - проверка email
- `POST /api/v1/auth/forgot-password` - сброс пароля
- `POST /api/v1/auth/reset-password` - установка нового пароля

**Users:**
- `GET /api/v1/users` - список пользователей
- `GET /api/v1/users/me` - текущий пользователь
- `POST /api/v1/users` - создание пользователя
- `PUT /api/v1/users/:id` - обновление
- `PATCH /api/v1/users/:id/change-group` - смена роли (CLIENT/VENDOR)

**Vendors & Clients:**
- CRUD операции для компаний

**Categories & Entries:**
- CRUD для справочников позиций смет

**Briefs:**
- CRUD для брифов (проектов)

**Offers:**
- CRUD для коммерческих предложений
- `GET /api/v1/offers/brief/:briefId` - все предложения по брифу

**Rates:**
- CRUD для тарифных карточек
- `POST /api/v1/rates/fork` - создание копии тарифа

**Зачем:**
- ✅ Фронтенд работает без единого изменения
- ✅ Полная обратная совместимость
- ✅ Все текущие функции сохраняются

### 3. Безопасность
**Задачи:**
- Реализация HMAC аутентификации (как в текущем NestJS)
- API Key fallback для разработки
- Role-based permissions (CLIENT/VENDOR/ADMIN)
- Timestamp validation (защита от replay атак)

**Зачем:**
- ✅ Безопасный доступ к API
- ✅ Разделение прав доступа
- ✅ Защита от атак

### 4. Deployment
**Задачи:**
- Настройка production и staging серверов
- CI/CD pipeline (GitHub Actions):
  - Автоматический build при push в main
  - Deploy на staging автоматически
  - Deploy на production по кнопке
- Nginx reverse proxy
- SSL сертификаты (Let's Encrypt)

**Зачем:**
- ✅ Автоматизированный deploy - экономия времени
- ✅ Безопасное HTTPS соединение
- ✅ Staging для тестирования перед production

### 5. File Storage
**Задачи:**
- Интеграция S3 (AWS/DigitalOcean Spaces) для файлов
- API endpoints для upload/download
- Валидация типов и размеров файлов

**Зачем:**
- ✅ Надежное хранилище файлов
- ✅ Масштабируемость
- ✅ Backup файлов

## Результат этапа:
🎯 **Работающий сайт на production серверах с новым Django backend**
- Текущий фронтенд работает без изменений
- Все существующие функции (создание брифов, estimation table, rate cards) работают
- CI/CD pipeline настроен и работает

# 🛠️ ЭТАП 1: КАБИНЕТ ВЕНДОРА (ДОРАБОТКА)

## Что будет сделано:

### 1. Custom Options в Estimation Table
**Задачи:**
- Backend: создание модели EntryOption для кастомных вариантов опций
- API endpoints для CRUD опций
- Frontend: UI для создания и выбора опций в estimation table
- Dropdown с вариантами в каждой строке

- Отображение выбранной опции в сводке
  
Вендор выбирает → цена позиции обновляется автоматически
```

### 2. Complex Units (например, часы × ролики)
**Задачи:**
- Backend: расширение модели Entry для поддержки сложных единиц измерения
- Логика перемножения единиц (например: 8 часов × 3 ролика = 24 единицы)
- Frontend: UI для ввода двух значений (количество и множитель)
- Автоматический расчет итоговых единиц и стоимости

**Пример:**
```
Позиция: Операторская работа
Цена: $100/час
Количество: 8 часов × 3 ролика = 24 единицы
Итого: $100 × 24 = $2,400
```

**Зачем:**
- ✅ Точный расчет для проектов с несколькими deliverables
- ✅ Прозрачность расчетов для клиента

### 3. Tax & Commission Calculations
**Задачи:**
- Backend: добавление полей tax_rate и commission_rate в модель Offer
- Автоматический расчет налогов и комиссий от итоговой суммы
- Frontend: поля для ввода % налога и комиссии
- Отображение в summary: Subtotal → Tax → Commission → Grand Total


### 4. Rate Cards (Тарифные карточки)
**Задачи:**
- CRUD операции для создания, редактирования, удаления тарифов
- Форкинг (копирование) тарифов для модификации
- Связывание тарифов с positions в estimation
- История изменений тарифов

**Зачем:**
- ✅ Быстрое заполнение estimation из заготовок
- ✅ Переиспользование стандартных цен
- ✅ Консистентность цен между проектами
- ✅ Легкое обновление цен

### 5. Templates (Шаблоны смет)
**Задачи:**
- Backend: модель OfferTemplate для хранения шаблонов
- API для сохранения текущей estimation как шаблон
- API для применения шаблона к новому offer
- Frontend: страница "Templates" со списком шаблонов
- Кнопка "Сохранить как шаблон" в estimation
- Кнопка "Применить шаблон" при создании

**Пример использования:**
```
1. Вендор создал estimation для "Рекламный ролик 30 сек"
2. Сохранил как шаблон "Стандартный ролик"
3. При новом проекте → применяет шаблон
4. Все категории и позиции загружаются автоматически
5. Вносит корректировки под конкретный проект
```

### 6. Excel Export
**Задачи:**
- Backend: генерация Excel файлов (библиотека openpyxl/xlsxwriter)
- Форматирование: таблица с категориями, позициями, ценами
- Два листа: "Estimation" (детальная таблица) + "Summary" (общая информация)
- API endpoint: `GET /api/v1/offers/:id/export/excel`
- Frontend: кнопка "Export to Excel"

### 7. Public Links (Публичные ссылки)
**Задачи:**
- Backend: модель PublicLink с UUID токеном
- Публичный endpoint (без авторизации) для просмотра offer
- Счетчик просмотров, опциональная дата истечения
- Frontend: кнопка "Share" в estimation
- Модальное окно с сгенерированной ссылкой и кнопкой "Copy"
- Публичная страница просмотра (read-only, красивый дизайн)
- Только client prices (internal prices скрыты)

**Пример ссылки:**
```
https://aivus.com/public/a7f3d9e2-4b5c-4a1e-8f9c-2d3e4f5a6b7c
```

**Зачем:**
- ✅ Быстрый шаринг сметы с клиентом
- ✅ Не требует регистрации клиента для просмотра
- ✅ Красивая презентация для клиента
- ✅ Отслеживание просмотров (analytics)

## Результат этапа:
🎯 **Полнофункциональный кабинет вендора с профессиональными инструментами**
- Вендор создает детальные сметы за 10-15 минут
- Поддержка сложных расчетов (опции, complex units, tax)
- Быстрое переиспользование типовых смет через templates
- Экспорт в Excel для клиентов
- Шаринг через публичные ссылки

# 💰 ВОЗМОЖНЫЕ РИСКИ

## 1. Технические риски (вероятность: средняя)
**Риск:** Проблемы совместимости API Django ↔ Next.js фронтенд

**Митигация:**
- Точное копирование структуры API из NestJS
- Тщательное тестирование каждого endpoint
- Возможность быстрого отката на NestJS если что-то не работает

## 2. Временные риски (вероятность: средняя)
**Риск:** Некоторые задачи могут занять больше времени чем планировалось

**Митигация:**
- Заложен буфер 3 недели
- Можно упростить некоторые фичи если необходимо
- Приоритизация критичных функций

## 3. AI Quality риски (вероятность: низкая)
**Риск:** AI может генерировать некорректные briefs или анализы

**Митигация:**
- Клиент всегда может редактировать результат AI
- Промпты будут тщательно протестированы
- Fallback на ручной ввод если AI не работает

## 4. Изменение требований (вероятность: высокая)
**Риск:** Новые требования появятся в процессе разработки

**Митигация:**
- Четкая фиксация MVP scope в начале
- Change request process для новых требований
- Новые фичи → backlog для версии 2.0
